# Requirement 03 - Asynchronous Queues

## الهدف

الهدف هو نقل المهام البطيئة التي لا يحتاج المستخدم لانتظارها إلى مسار خلفي Background Processing.

في نظام التجارة الإلكترونية، بعض المهام مثل إرسال الفاتورة أو الإشعار أو إشعار المستودع لا يجب أن تكون داخل المسار الرئيسي للطلب، لأن المستخدم يحتاج فقط لمعرفة أن الطلب تم استلامه بنجاح.

---

## المشكلة قبل الحل

في النسخة المتزامنة:

```http
POST /orders/before-sync
```

الطلب ينفذ:

1. إنشاء الطلب.
2. انتظار إرسال الإشعار أو الفاتورة.
3. إرجاع الرد للمستخدم بعد انتهاء كل شيء.

تمت محاكاة إنشاء الطلب بزمن 300ms، وتمت محاكاة إرسال الإشعار بزمن 3000ms. لذلك زمن استجابة المستخدم يكون حوالي 3300ms.

هذا حل صحيح وظيفياً، لكنه غير مناسب تحت الضغط لأن كل مستخدم ينتظر مهمة ثانوية بطيئة.

---

## الحلول المقترحة

### 1. إبقاء المعالجة متزامنة

هذا أبسط حل، لكنه يجعل زمن الاستجابة عالياً ويستهلك Threads لفترة أطول.

### 2. استخدام Queue خارجية مثل RabbitMQ أو Kafka

هذا حل إنتاجي قوي جداً، لكنه يزيد التعقيد ويتطلب تشغيل خدمة إضافية.

### 3. استخدام In-Process Queue مع BackgroundService

هذا هو الحل المختار للمشروع الحالي لأنه يثبت فكرة المعالجة غير المتزامنة بوضوح دون إضافة بنية خارجية ثقيلة.

---

## الحل المختار

تم استخدام:

```csharp
Channel<NotificationJob>
```

داخل:

```text
Services/NotificationQueueService.cs
```

وتمت معالجة الرسائل بالخلفية داخل:

```text
Services/NotificationWorkerService.cs
```

باستخدام:

```csharp
BackgroundService
```

كما تم جعل الـ Channel محدوداً بسعة 200 Job حتى لا يتحول إلى Queue غير محدودة قد تستهلك الذاكرة تحت الضغط.

---

## بعد الحل

في النسخة غير المتزامنة:

```http
POST /orders/after-async
```

الطلب ينفذ:

1. إنشاء الطلب.
2. إنشاء NotificationJob.
3. وضع الـ Job داخل Queue.
4. إرجاع الرد للمستخدم مباشرة.
5. معالجة الإشعار لاحقاً داخل Background Worker.

زمن استجابة المستخدم يصبح تقريباً 300ms بدل 3300ms، لأن مهمة الإشعار لم تعد داخل مسار الطلب الرئيسي.

---

## Endpoints المستخدمة للاختبار

```http
POST /notifications/reset
```

لتصفير قياسات الـ Queue قبل الاختبار.

```http
POST /orders/before-sync
```

لاختبار الحالة المتزامنة قبل الحل.

```http
POST /orders/after-async
```

لاختبار الحالة غير المتزامنة بعد الحل.

```http
GET /notifications/status
```

لمراقبة حالة الـ Queue وعدد الرسائل التي تم إدخالها ومعالجتها.

---

## ماذا نراقب؟

من `/notifications/status` نراقب:

- `queuedJobs`: عدد المهام التي دخلت إلى الـ Queue.
- `processedJobs`: عدد المهام التي عالجها الـ Background Worker.
- `currentQueueDepth`: عدد المهام التي ما زالت تنتظر داخل الـ Queue.
- `activeBackgroundWorkers`: هل يوجد Worker يعالج حالياً.
- `maxQueueDepthObserved`: أعلى عمق وصلت له الـ Queue أثناء الاختبار.

---

## الإثبات المتوقع

قبل الحل:

```text
/orders/before-sync
```

زمن الاستجابة يكون قريباً من 3300ms لأن المستخدم ينتظر الإشعار.

بعد الحل:

```text
/orders/after-async
```

زمن الاستجابة يكون قريباً من 300ms لأن المستخدم لا ينتظر الإشعار، بينما تظهر المهمة لاحقاً في `/notifications/status` ويتم معالجتها بالخلفية.

## الخلاصة

تم تحقيق المتطلب الثالث لأن النظام صار يفصل بين المسار الرئيسي للطلب والمهام الثانوية البطيئة.

النتيجة الهندسية:

- قبل الحل: المستخدم ينتظر مهمة الإشعار البطيئة.
- بعد الحل: يتم إرجاع الرد بسرعة، وتتم معالجة الإشعار بالخلفية.
- الـ Queue محدودة السعة حتى لا تستهلك الذاكرة بشكل غير مضبوط.
- يمكن مراقبة حالة الـ Queue عبر endpoint مستقل.
